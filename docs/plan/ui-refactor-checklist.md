# UI 改版清单（参考案例版）

> 参考案例：用户提供的 OmniFocus 风格深色界面截图。
> 目标：让 Recall 的整体界面从“功能可用”升级为“结构清晰、层级明确、专业工具感强”的产品界面。

## 一、改版目标

本轮 UI 改版不是简单换皮，而是完成以下 4 件事：

1. 统一整体视觉语言
2. 拉开页面层级，让用户一眼知道哪里是导航、哪里是内容、哪里是操作
3. 提高信息密度下的可读性，做到“内容多但不乱”
4. 让 Recall 的气质更接近专业效率工具，而不是零散面板拼接

## 二、参考案例要点拆解

基于截图，建议重点吸收以下特征：

### 1) 深色主题统一且克制
- 背景、侧栏、主内容区、悬浮层有明确明暗层级
- 不依赖过多高饱和颜色制造“高级感”
- 强调靠结构、留白、对比，而不是炫技特效

### 2) 左侧导航信息架构清晰
- 导航区一眼可分：主入口 / 分组 / 特殊视图 / 标签体系
- 图标、文字、计数、选中态表达明确
- 导航本身就是“结构索引”，不是简单按钮堆叠

### 3) 主内容区阅读路径顺畅
- 标题、说明、列表主体、次级说明有明确顺序
- 用户不需要扫很多地方才能找到重点
- 当前上下文（在哪个项目/视图中）必须足够明显

### 4) 列表层级很强
- 父项、子项、分组、状态之间的关系一眼能看懂
- 不是靠花哨边框，而是靠缩进、字重、颜色、间距、图标共同表达

### 5) 信息密度高但不压迫
- 同一屏可以容纳较多信息
- 但通过排版节奏控制，让界面依然有呼吸感

## 三、Recall UI 改版任务清单

## A. 全局视觉层

### A1. 统一深色主题层级
- [ ] 重新梳理页面背景、侧栏背景、主面板背景、次级面板背景
- [ ] 统一边框强弱规则（强边框 / 软边框 / 无边框）
- [ ] 统一文字层级（标题 / 正文 / 次级信息 / 禁用态）
- [ ] 控制强调色使用范围，避免满屏都在抢视觉焦点

**DoD：** 任意 3 个核心页面放在一起看，背景层级、边框语言、文字层级一致。

### A2. 建立“专业工具感”视觉基线
- [ ] 减少廉价渐变、随机阴影、随意高亮
- [ ] 强化克制、稳定、长期使用不疲劳的风格
- [ ] 保证深色模式下的对比度够用，但不过曝

**DoD：** 整体气质统一，不再像多个不同风格模块拼接。

## B. 信息架构与页面骨架

### B1. 重构左侧导航
- [ ] 重新定义导航分区：主页面 / 任务视图 / 时间视图 / AI / 设置
- [ ] 为导航增加明确的分组标题或视觉区隔
- [ ] 统一图标、文字、计数、选中态样式
- [ ] 检查折叠/展开后的可理解性

**DoD：** 用户首次进入时，能在几秒内理解导航结构与当前位置。

### B2. 重构页面 Header 区
- [ ] 统一页面标题、说明、副标题、快捷操作布局
- [ ] 当前视图名称要足够突出
- [ ] 主要操作与次要操作区分清楚

**DoD：** 页面顶部不再杂乱，标题区一眼能看懂“这里是什么、现在能做什么”。

### B3. 重构主内容区骨架
- [ ] 统一页面结构：Header / Summary / Controls / Content
- [ ] 减少内容区块随意堆叠
- [ ] 保持筛选、统计、主列表之间的层级顺序稳定

**DoD：** 不同页面切换后，结构认知成本低。

## C. 列表 / 卡片 / 层级表达

### C1. 重做任务列表层级
- [ ] 明确父任务、子任务、分组标题、状态标签的视觉差异
- [ ] 用缩进、字重、颜色、间距表达层级，而不是只靠小字或一条线
- [ ] 检查完成态、进行中、延期、重点项的识别效率

**DoD：** 列表在信息量较大时，仍然能快速扫读。

### C2. 重做卡片信息排布
- [ ] 统一卡片内标题、元信息、操作区、状态区顺序
- [ ] 卡片减少“所有信息都一样重”的问题
- [ ] 重要信息前置，低优先信息后置

**DoD：** 卡片内容一眼能抓到重点，不需要逐行阅读才能理解。

### C3. 优化多层级内容的展开/收起体验
- [ ] 展开区域和主内容区之间建立清晰从属关系
- [ ] 动效服务于理解层级，不做廉价淡入淡出
- [ ] 展开后布局不跳、不乱、不挤压重点

**DoD：** 展开/收起不会打断用户对上下文的理解。

### C4. 增加检查功能页面（Review / 检查视图）
- [ ] 新增“检查 / Review”页面入口，并在导航中占据明确位置
- [ ] 支持按项目 / 列表 / 时间周期查看待检查项
- [ ] 页面结构参考案例：左侧导航、主检查列表、右侧详情/属性区
- [ ] 检查项支持状态、标签、开始时间、计划时间、截止时间等信息展示
- [ ] 支持逐项检查、批量跳过、回到待处理上下文

**DoD：** 用户可以把“检查”当成独立工作流使用，而不是临时拼出来的筛选页。

### C5. 增加附近页面（Nearby / 附近可完成任务）
- [ ] 新增“附近 / Nearby”页面入口，并在导航中有独立位置
- [ ] 支持基于定位或地点标签筛出“当前附近适合完成”的任务
- [ ] 支持展示地点、距离、可执行条件、任务优先级
- [ ] 支持按“最近 / 最紧急 / 最适合顺路完成”排序
- [ ] 支持从附近页面直接进入任务详情或标记完成

**DoD：** 用户到某个位置时，可以快速看到“现在在附近最值得顺手完成的任务”，而不是自己手动翻列表。

## D. 关键页面改版优先级

### 第一批（优先做）
- [ ] 任务主页面
- [ ] 时间轴页面
- [ ] 左侧导航 / 全局布局框架

**原因：** 这是用户最高频接触区域，决定第一印象和整体效率感。

### 第二批
- [ ] AI 面板
- [ ] 项目/标签相关页面
- [ ] 检查功能页面（Review / 检查视图）
- [ ] 附近页面（Nearby / 附近可完成任务）
- [ ] 设置页

**原因：** 这些页面影响整体一致性，但不一定是最高频主入口。

### 第三批
- [ ] 空态 / 错态 / 加载态统一
- [ ] 弹窗 / 抽屉 / 浮层统一
- [ ] 移动端细节回收

**原因：** 用于补齐体验闭环，防止主页面改好了，细节还在掉链子。

## E. 组件级统一清单

### E1. 优先统一的组件原语
- [ ] 侧栏导航项
- [ ] 页面标题区
- [ ] 主卡片 / 次卡片
- [ ] 按钮体系
- [ ] 状态标签（badge）
- [ ] 筛选栏 / 工具栏
- [ ] 列表项 / 子项

**DoD：** 新页面优先复用这些原语，不再重复造样式。

### E2. 状态反馈统一
- [ ] 统一 hover / active / selected / focus-visible
- [ ] 统一成功 / 警告 / 错误 / 禁用态
- [ ] 统一加载中 / 空内容 / 失败重试的表达方式

**DoD：** 全站状态反馈是同一种语言，而不是每个模块各说各话。

### E3. 新建任务入口改为卡片式创建
- [ ] 参考用户提供的案例，将“新建任务”从直接裸露输入/按钮样式，调整为更完整的卡片式创建入口
- [ ] 新建卡片需容纳标题、备注、日期、优先级、标签等常用信息入口
- [ ] 创建态需要和现有列表/卡片视觉统一，不能像临时插入的编辑条
- [ ] 去掉当前“直接新建”的生硬样式，增强创建时的结构感与专业感

**DoD：** 用户在创建任务时，感受到的是一个完整的任务卡片编辑入口，而不是一条突兀的临时输入区域。

## F. 可执行的下一步建议

### 第一阶段：先出结构，不急着抠细节
- [ ] 先列出全部核心页面
- [ ] 给每个页面标注：当前问题、目标形态、优先级
- [ ] 确定第一批改版页面

### 第二阶段：先统一骨架和原语
- [ ] 先做导航、Header、卡片、列表项四类基础原语
- [ ] 再回到各页面做替换

### 第三阶段：补视觉细节与动效
- [ ] 在结构稳定后再统一动效、阴影、边框、密度
- [ ] 避免边做结构边频繁改视觉，导致返工

## 四、验收标准

完成这一轮 UI 改版后，应达到：

- 用户第一次打开就能快速理解界面结构
- 高信息密度页面依然好读
- 左侧导航、页面标题、主内容区关系明确
- 深色主题统一、克制、有专业工具感
- 关键页面之间像同一个产品，不像不同阶段拼出来的页面
