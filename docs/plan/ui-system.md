# UI 规范系统（提案）

> 目标：建立一套可复用、可扩展、可审计的 UI 规范，让后续“加功能 / 改界面”优先走规范，而不是每次局部拼样式。

## 为什么现在要做

Recall 已经有：
- theme token（accent / gradient）
- glass panel / card / timeline 等若干视觉模式
- 多个功能面板（任务、时间轴、AI、设置、倒计时、习惯）

但还缺：
- 统一的组件层级规范
- 统一的间距 / 圆角 / 边框 / 状态色规则
- 页面扩展时的“该怎么做”流程

这会导致后续扩功能时：
- 容易重复造样式
- 新旧界面观感不统一
- 小修一处牵一片

## UI 系统应包含什么

### 1) Design Tokens
- 颜色 token（文本 / 背景 / 边框 / 状态 / 强调）
- 渐变 token
- 间距 token
- 圆角 token
- 阴影 token
- 动效 token

### 2) 组件规范
- 面板（panel / glass-panel / soft-panel）
- 卡片（信息卡 / 列表卡 / 可点击卡）
- 筛选条 / toolbar
- 标签 badge / 状态 badge
- 输入框 / 下拉 / 按钮
- 空状态 / 错误状态 / 加载状态

### 3) 页面布局规范
- 页面头部层级
- 主内容区宽度与栅格
- 多模块组合方式
- 响应式断点策略

### 4) 功能扩增流程
以后新增功能时，优先按下面顺序：
1. 先选已有 token
2. 再选已有组件模式
3. 再决定页面放置位置与层级
4. 只有现有规范不够时，才新增新的 UI 原语

## 建议产出物

- `docs/plan/ui-system.md`（本文件，原则）
- `docs/plan/ui-spec.md`（正式规范：token / 组件 / 页面）
- `src/app/globals.css` / 组件层：逐步收敛到 token 化命名
- 示例页面改造：至少 1~2 个典型页面按规范回收

## 优先级建议

### 高
- 统一 token 命名与使用边界
- 统一 card / panel / toolbar / badge 规范
- 制定“新增功能先走 UI 规范”的约束

### 中
- 整理页面栅格与响应式规则
- 整理空状态 / 错误状态 / 加载状态

### 低
- 组件目录进一步分层
- 为设计规范补截图示例
