# Recall Agent 产品迭代工作流

> 目的：让 agent 团队不只是“完成任务”，而是能稳定推进产品迭代。
> 该文档建立在 `docs/plan/collaboration-workflow.md` v2 之上，补充产品视角的迭代规则。

## 1. 核心原则

### 1.1 不是只做事，而是推进产品
每一轮都必须回答：
1. 这一轮提升的是哪个产品面？
2. 这一轮只解决哪一个最小问题？
3. 做完后，用户能感知到什么变化？
4. 这轮完成定义是什么？
5. 下一轮最自然接什么？

没有这 5 个答案，就不算完整的产品迭代轮。

### 1.2 技术闭环 != 产品闭环
`build / audit / commit / push` 只是技术交付闭环。
真正的产品闭环还必须回答：
- 用户有没有更容易完成某件事？
- 页面 / 功能的完成度有没有提升？
- 下一轮是不是更自然了？

## 2. 团队角色升级

### `main`
- 决定当前版本最该推进的产品面
- 判断某一轮是否值得开跑
- 在技术闭环之外，额外判断“这轮是否真的推动了产品”

### `pm-coder`
- 不只拆任务，还要维持版本连续性
- 每轮开始前产出“小目标卡”
- 每轮结束后必须补一条：**建议下一轮做什么，为什么**

### `code-helper`
- 聚焦实现当前最小产品增量
- 禁止把“顺手能做”的东西混进当前轮
- 交付时必须说明：这轮用户感知变化是什么

### `ui-designer`
- 不只是审美点评，而是产品体验位
- 固定输出：
  - **必须改**：不改会影响本轮完成度
  - **建议改**：改了明显更好，但不一定本轮做
  - **可暂缓**：先别扩题

### `qa-tester`
- 采用双结论输出：
  1. **技术验收**：build / 回归 / 风险
  2. **产品验收**：这轮目标是否真的达成

## 3. 每轮必须包含“产品收益句”

每一轮正式开跑前，除小目标卡外，还必须补一条：

```md
### 产品收益句
本轮完成后，用户将更容易 ________。
```

示例：
- 本轮完成后，用户将更容易在四象限页理解任务堆积位置，并完成拖拽归类。
- 本轮完成后，用户将更容易在 Review 页判断当前是“加载中”、“无待检查项”还是“本轮完成”。

如果写不出这句话，说明这轮目标还不够产品化。

## 4. 每轮结束时必须补“下一轮建议”

结束态除了汇报当前结果，还必须补：

```md
### 下一轮建议
- 建议下一轮做：
- 原因：
```

作用：
- 避免团队每轮都重新找方向
- 让版本推进从“单轮任务”变成“连续迭代”
- 让 `pm-coder` 承担版本连续性，而不是只做一次性调度

## 5. QA 输出改成双结论

以后 QA 的标准输出结构：

### 技术验收
- 是否 build 通过
- 是否存在阻断问题
- 当前交付状态：可提交 / 可推送 / 已推送

### 产品验收
- 本轮定义的小目标是否真的达成
- 用户是否能感知到改动价值
- 是否存在“能编译但没有明显变好”的情况

## 6. 推荐迭代节奏

每一轮固定四步：
1. **选本轮目标**：从当前版本主线里选一个最值得做的小点
2. **落一个小步**：只推进这一点
3. **双重验收**：技术验收 + 产品验收
4. **决定下一轮**：明确下一步接什么

## 7. 当前适用于 Recall 的工作方式

当前建议：
- `collaboration-workflow.md` 负责协作秩序
- 本文负责产品迭代方向
- 两份文档配合使用：
  - 一个防止团队跑散
  - 一个保证团队不是只会做任务，而是真的推进产品

## 8. 当前落地要求（从下一轮开始生效）

从下一轮开始，所有多 agent 任务都应补齐以下 4 项：
1. 小目标卡
2. 产品收益句
3. QA 双结论
4. 下一轮建议

缺任何一项，都不能算“高质量完成”。
