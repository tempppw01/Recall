# 历史版本与当前计划规则说明

> 目的：解释“仓库历史 tag 已到 v0.3.1，但当前计划流程要求从 v0.1 单线推进”之间的关系，避免后续执行、审计和发版时产生歧义。

## 1. 当前事实

当前仓库中已经存在以下历史发布：

- `v0.2.0`
- `v0.2.1`
- `v0.3.0`
- `v0.3.1`

对应的 GitHub Release 与 release notes 也已存在，属于既成事实，不在未获明确授权的情况下进行回滚、删 tag 或重写历史。

## 2. 当前执行规则

自本轮规则收敛后，计划体系以 `docs/plan/versions/v0.1.md` 作为唯一当前执行入口：

- 当前工作版本固定为 `v0.1`
- `v0.2` / `v0.3` / `v1.0` 仅保留规划，不允许开工
- 每轮变更后必须补审计报告，并同步更新 `docs/plan/README.md`

这是一套“从现在开始严格执行”的计划规则，不等于自动抹除历史上已经发生的发布事实。

## 3. 解释口径

后续所有计划、审计、阶段报告统一采用以下口径：

1. **历史 tag 保留为历史发布记录**
   - 继续在 `CHANGELOG` 和 GitHub Releases 中保留。
   - 不把它们当作当前计划执行顺序的依据。

2. **当前计划顺序按文档规则重新收口**
   - 当前执行入口只认 `docs/plan/versions/v0.1.md`。
   - 只有当 v0.1 退出条件被明确满足后，才允许切换到 `v0.2` 计划文件。

3. **发版前必须显式检查并说明冲突**
   - 若未来需要继续沿用既有 tag 序列，则必须在发版说明中注明：
     - “计划版本顺序”与“历史 tag 序列”并不完全一致；
     - 当前计划文档用于治理执行顺序，历史 tag 用于保留既有发布事实。

## 4. 对退出条件的影响

对于 `v0.1`：

- `docs/plan/versions/v0.1.md` 已全部 ✅
- 最近一轮审计无未处理高危问题
- 但“CHANGELOG 与 Release notes 一致”只能说明历史发布记录完整，**不能自动证明当前流程意义上的 `v0.1.0` 已按新规则重新发布**

因此，当前可以认为：

- **v0.1 的任务层面已完成**
- **v0.1 的历史发布口径仍需人工确认**

## 5. 后续建议

### 方案 A：保留历史 tag，不补发 v0.1.0（推荐）
- 视 `v0.2.0` 为“历史上承接 v0.1 完成后的既成发布”
- 后续计划仍按 `v0.1 -> v0.2 -> ...` 管理，但发版时单独注明“历史 tag 已存在”
- 优点：不改历史、风险最低

### 方案 B：额外补一个说明性 v0.1.0 tag（需明确授权）
- 仅在维护者明确要求下执行
- 风险：会让时间线和既有 `v0.2.x/v0.3.x` 语义更混乱

### 方案 C：重写历史 tag / Release（不推荐，且需明确授权）
- 包括删 tag、重打 Release、重写 changelog
- 风险最高，默认不执行

## 6. 默认决策

在没有额外指令前，默认采用 **方案 A**：

- 保留历史 tag / Release 不动
- 当前计划治理继续以 `v0.1` 为唯一执行入口
- 后续阶段报告与发版说明中显式注明“历史 tag 与当前规则并存”的事实
