## 1) AI 管理助手：推荐“直接操作”（Phase 2 补完）

目标：管理助手在给出 recommendations 的同时，支持用户一键执行建议动作（置顶/改优先级/改日期）。

- [x] 定义动作协议（后端返回字段规范 + 前端执行策略）

  约定（manage-agent recommendations 单条结构）：
  - id: string（必须存在于 tasks）
  - title: string（展示用，可与任务标题一致）
  - reason: string（可选，展示用）
  - suggestedPriority: 0|1|2（可选；若提供则前端提供“一键改优先级”）
  - suggestedPinned: boolean（可选；若提供则前端提供“一键置顶/取消置顶”）
  - suggestedDuePreset: 'today'|'tomorrow'|'tonight'（可选；若提供则前端提供“一键改日期”）

  执行策略（前端）：
  - 所有动作都走现有本地更新函数（updateTask / toggleTaskPinned / quickSetPriority / quickSetDuePreset / updateTaskDueDate）。
  - 动作执行后：刷新 tasks，并清理该 recommendation 的 action 状态（可选）。
  - 若 recommendation 同时给出多种 suggested 字段，前端可分别提供按钮（用户点哪个就执行哪个）。

  回退：如果某字段缺失或非法，隐藏对应操作按钮。
- [x] 后端 `manage-agent` 输出扩展：支持 `suggestedPinned` / `suggestedDuePreset`（today/tomorrow/tonight）
- [x] 前端推荐卡片增加“直接操作”按钮：
  - [x] 一键置顶/取消置顶
  - [x] 一键改优先级（使用 suggestedPriority）
  - [x] 一键改日期（使用 suggestedDuePreset）
- [x] 推荐执行后：本地状态 +（若配置）PG 同步都正确
- [x] 文档：更新 AI 助手说明（README 或 docs）

## 2) Prisma / 数据库边界收敛（0.2.0 前置）

目标：收敛动态 PrismaClient / x-pg-* 方案，明确安全边界与性能边界。

- [x] 审查 `src/lib/prisma.ts` 动态 client 创建策略（风险、并发、生命周期）
- [x] 决定 `x-pg-*` 请求头方案：保留（默认关闭）+ 降级（仅未登录、且强制开关）+ 文档化
- [x] 限制 PrismaClient 实例创建策略（避免连接池耗尽；必要时增加缓存/单例/守卫）

- [x] 增加 DB 链路验证脚本（可快速验证 DATABASE_URL/迁移/连通性）
- [x] 增加最小 migration / 初始化文档（prisma migrate / deploy）
- [x] 增加常见部署场景说明（Docker、本地、远程 PG）

改动点（本阶段）：
- 后端：统一 request DB 上下文解析（session 优先；仅未登录才允许 dynamic PG），避免不同 API 路由行为不一致（尤其 DELETE）。
- 动态 PG：为连接串补充 `connection_limit/connect_timeout/pool_timeout` 参数，并保留缓存 + TTL 回收，降低连接池耗尽风险。
- 文档/脚本：新增 `docs/database.md` + `scripts/db-check.sh`。

审计发现与处理：
- ✅ 修复：部分 DELETE 路由存在“动态 PG 优先于 session”的不一致风险（潜在越权/混淆边界）。已统一收口。
- ✅ 修复：动态 client 未显式限制连接参数，容易在错误配置/抖动时拖垮服务。已加守卫参数（可通过 env 调整）。

风险项/技术债：
- 当前仓库未纳入 Prisma migrations（仅 schema）。若要生产可追溯发布，需要将 `prisma/migrations` 纳入版本控制，并明确 migrate 策略。

产品优化建议：
- 高优先级：在设置页明确提示“动态 PG 模式的风险”，并在 UI 上引导设置 `PG_HEADERS_TOKEN` 与 host allowlist。
- 中优先级：把“数据库连接测试 / db-check”做成可复制的一键命令（含诊断输出），降低自部署门槛。
