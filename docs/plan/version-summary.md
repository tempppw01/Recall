# Recall 版本规划总表（0.0.1 ~ 0.0.4）

> 这份总表用于快速查看当前计划主线。
> 详细任务仍以 `docs/plan/versions/*.md` 为准；本文件负责给出“每个版本到底干什么”的总览。

## 一、版本总览

| 版本 | 状态 | 核心目标 | 关键词 |
|---|---|---|---|
| 0.0.1 | 已完成 | 打稳 MVP 基线 | 稳定化、边界收敛、发布流程 |
| 0.0.2 | 执行中（接近收口） | 收敛结构与视觉体验 | 导航、Header、卡片、深色主题 |
| 0.0.3 | 仅规划 | 增强功能视图能力 | 检查页、附近页、上下文任务 |
| 0.0.4 | 仅规划 | 完成整体 UI / 界面重构 | UI 系统化、专业工具感、统一体验 |

## 二、各版本说明

## 0.0.1：MVP 稳定化

### 目标
让 Recall 达到“核心流程可用、关键边界收敛、可以可靠部署”的状态。

### 已完成内容
- AI 管理助手“直接操作”闭环
- 动态 PG / Prisma 边界收敛
- 时间轴 Phase 2 体验增强
- 计划目录收口
- 审计与发布流程基础闭环

### 当前判断
这是当前计划体系里的**已完成基线版本**。
后续所有规划，默认都建立在 0.0.1 已完成之上。

## 0.0.2：体验增强 + 结构整理

### 目标
先不急着上更多复杂功能，先把现有产品从“能用”整理到“顺眼、顺手、结构清楚”。

### 重点任务
- 左侧导航层级优化（已完成第一轮）
- 页面 Header 重构（已完成第一轮）
- 主内容区骨架优化（已完成第一轮）
- 列表 / 卡片信息排布优化（已完成核心区域）
- 新建任务改为卡片式创建（已完成第一轮）
- 深色主题统一（持续收敛中）
- 设置 / modal / 辅助面板统一（已完成核心范围）
- 最小健康检查与 migrations 收口

### 这一版要解决的问题
- 页面结构还不够统一
- 新建任务样式太直接、太生硬
- 界面的专业感、层级感还不够稳定

### 预期结果
用户打开后，会明显感觉：
- 页面更有秩序
- 信息更好扫读
- 新建 / 浏览 / 操作都更顺手

## 0.0.3：功能视图增强

### 目标
增加更有上下文感的任务视图，让 Recall 变得更像一个完整效率系统，而不只是基础任务清单。

### 重点任务
- 检查功能页面（Review）
- 附近页面（Nearby）
- 更强的任务上下文视图
- sync 模型与冲突策略明确化
- 错误码 / requestId / 结构化日志收敛
- 深度健康检查与同步诊断增强

### 这一版要解决的问题
- 当前任务视图还不够丰富
- 缺少“检查”“附近可做”这类高价值工作流
- 同步问题定位成本还偏高

### 预期结果
用户可以更自然地按“场景”完成任务，而不是只在一个大列表里翻来翻去。

## 0.0.4：整体 UI / 界面重构

### 目标
完成整体 UI / 界面重构，把 Recall 收敛成统一、成熟、专业的效率工具体验。

### 重点任务
- 建立统一 UI 规范系统
- 统一关键页面 UI 原语
- 整体 UI / 界面重构
- 新增功能优先复用 UI 规范
- 统一关键交互动效系统
- 补齐稳定发布要求

### 这一版要解决的问题
- 各页面虽然可能都能用，但还不够像“同一个产品”
- 视觉语言、组件语言、状态反馈、动效语言需要系统统一

### 预期结果
Recall 会从“逐步补功能的产品”进入“整体体验成熟的产品”阶段。

## 三、你当前新增的需求，放在哪一版

### 已归入 0.0.2
- 左侧导航优化
- 主内容区结构优化
- 列表层级优化
- 深色主题统一
- 页面 Header 重构
- 卡片信息排布优化
- 新建任务改成卡片式创建
- 去掉原来直接新建的生硬样式

### 已归入 0.0.3
- 检查功能页面（Review）
- 附近页面（Nearby）
- 更强的上下文任务视图

### 已归入 0.0.4
- 整体 UI / 界面重构
- 专业工具感统一
- 全产品视觉语言统一
- 页面与组件系统级收敛

## 四、当前建议的推进顺序

1. **先守住 0.0.1 作为稳定基线**
2. **先做 0.0.2，把结构和视觉收顺**
3. **再做 0.0.3，把检查页 / 附近页这类高价值视图补上**
4. **最后做 0.0.4，把整个产品统一成成熟版本**

## 五、一句话总结

- `0.0.1`：先稳住
- `0.0.2`：先收顺
- `0.0.3`：再补强
- `0.0.4`：再统一
