# Recall 多 Agent 小步推进流程 v2

> 目的：避免单个 agent 长时间独跑、反复 restart、卡在“读上下文/重写同一文件”，把推进改成可监控、可回滚、可交付的小步协作流程。

## 1. 角色分工

- `main`
  - 对外同步进度
  - 决定是否中断 / 调整 / 升级处理方式
  - 负责中途校准与必要时的主控接管
  - 统一使用“可提交 / 可推送 / 已推送”三段状态口径
- `pm-coder`
  - 只负责拆解任务、定义本轮目标、安排顺序、整合结果
  - 默认**不直接长时间写代码**
  - 每轮必须先产出“小目标卡”
- `code-helper`
  - 负责具体代码实现与局部修复
  - 只落当前小步，不顺手扩题
- `ui-designer`
  - 负责 UI / 交互检查、视觉一致性、信息层级反馈
  - 默认输出：**必须改 / 建议改 / 可暂缓**三级结论
- `qa-tester`
  - 分两段工作：
    1. 验收准备：提前整理验收关注点，不抢跑 build
    2. 最终验收：代码落地后再跑 build / 回归，给出可否推送结论

## 2. 基本规则

### 2.1 一次只推进一个小步
每一轮只允许处理一个明确的小目标，例如：
- Review 页新增一个视图切换
- 修一个状态流问题
- 收口一个 panel 的 UI

禁止把多个页面 / 多个主题打包成一轮大杂烩。

### 2.2 小步 Git
每个小步必须走完整闭环：
1. 改代码 / 文档
2. build / 验证（纯文档轮需说明为何无需 build）
3. 写 audit（放在 `docs/plan/audits/`）
4. git commit
5. git push

没有完成这 5 步，不能算“好了”。

### 2.3 pm-coder 不得闷头全包
`pm-coder` 的默认动作应该是：
1. 定义小目标
2. 指派给合适 agent
3. 收结果
4. 决定下一小步

只有在任务极小、且明显不值得分发时，才允许自己直接动手。

### 2.4 避免反复 restart 同一个 agent
如果需要补充要求，优先：
- 看当前结果
- 基于结果定义下一个小步

不要通过反复 restart 把 agent 拉回“重新读上下文”的起点。

## 3. 每轮开始前必须有“小目标卡”

每轮正式开跑前，必须先写清楚下面 4 件事：

1. **这轮只做什么**
2. **这轮明确不做什么**
3. **谁负责什么**
4. **验收标准是什么**

推荐模板：

```md
### 小目标卡
- 本轮目标：
- 本轮不做：
- 分工：
  - pm-coder：
  - code-helper：
  - ui-designer：
  - qa-tester：
- 验收标准：
```

没有小目标卡，不进入正式执行。

## 4. 标准执行链路

### 模式 A：正常多 agent 协作
1. `pm-coder` 定义一个小步目标，并产出小目标卡
2. `pm-coder` 拉起 `code-helper` 落代码
3. 如涉及明显 UI 变化，再拉 `ui-designer` 做检查
4. `qa-tester` 先做验收准备，代码完成后再做最终验收
5. `pm-coder` 汇总结果
6. 完成 commit / push
7. `main` 对外汇报

### 模式 B：主控接管
如果发现以下任一情况，`main` 可直接接管当前轮：
- agent 被反复 restart，持续不出结果
- 多 agent 没有真正分发，只停留在口头协作
- 已经改了代码，但迟迟不 build / 不提交
- 任务规模很小，继续调度成本高于直接收尾
- 中途校准时确认 agent 已明显偏题

接管后仍要遵守：
- 小步目标
- build / 验证
- audit
- commit
- push

## 5. 增加固定“中途校准点”

每轮除开始态和结束态外，必须至少有一次主控中途校准，检查这 4 件事：

1. 代码 / 文档到底有没有真实落下去
2. build / 验证是否已经开始
3. 当前实现是否偏题
4. 是否需要主控接管

中途校准后只能进入 3 种状态之一：
- **继续**：方向正确，继续本轮
- **改向**：目标不变，但实现方式要调整
- **接管**：停止分发，由 `main` 直接收口

## 6. 实时汇报规则

每个小步至少汇报 3 次：
1. **开始时**：这一步做什么，谁来做
2. **中间态**：代码是否已改、build 是否开始、当前是继续 / 改向 / 接管中的哪一种
3. **结束时**：是否通过、commit 是什么、当前属于可提交 / 可推送 / 已推送中的哪一种

禁止只在最后一句说“做完了”。

## 7. 统一交付状态口径

以后对外只允许使用以下 3 个状态词：

- **可提交**：代码 / 文档已改完，验证通过，适合进入 commit
- **可推送**：commit 已完成，当前适合 push 到远程
- **已推送**：远程分支已确认更新

禁止把“可推送”和“已推送”混着说。

## 8. 当前适用于 Recall 的推荐拓扑

- `pm-coder` → `code-helper`
- `pm-coder` → `ui-designer`
- `pm-coder` → `qa-tester`
- `main` 监控全部状态

当前不建议一上来开放所有 agent 全互调，避免任务流失控。

## 9. 当前问题复盘（2026-03-21）

本次流程卡住的直接原因：
- `pm-coder` 实际仍在单兵写代码
- 多次 steer 触发 restart，导致它反复回到“重读上下文 / 重写 ReviewPanel”的起点
- 已写文件，但没有稳定推进到 build / audit / commit / push
- 对外汇报时一度把“可 push”和“已 push”混在一起，造成误判

后续必须按本文件 v2 流程执行，避免再次出现“代码改了但流程没收口”的假完成状态。
