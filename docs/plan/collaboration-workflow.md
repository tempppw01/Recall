# Recall 多 Agent 小步推进流程

> 目的：避免单个 agent 长时间独跑、反复 restart、卡在“读上下文/重写同一文件”，把推进改成可监控、可回滚、可交付的小步协作流程。

## 1. 角色分工

- `main`
  - 对外同步进度
  - 决定是否中断/调整/升级处理方式
  - 监控 agent 是否真的按流程执行
- `pm-coder`
  - 只负责拆解任务、定义本轮目标、安排顺序、整合结果
  - 默认**不直接长时间写代码**
- `code-helper`
  - 负责具体代码实现与局部修复
- `ui-designer`
  - 负责 UI/交互检查、视觉一致性、信息层级反馈
- `qa-tester`
  - 负责 `npm run build`、基础回归、验收结论

## 2. 基本规则

### 2.1 一次只推进一个小步
每一轮只允许处理一个明确的小目标，例如：
- Review 页新增一个视图切换
- 修一个状态流问题
- 收口一个 panel 的 UI

禁止把多个页面/多个主题打包成一轮大杂烩。

### 2.2 小步 Git
每个小步必须走完整闭环：
1. 改代码
2. build / 验证
3. 写 audit（放在 `docs/plan/audits/`）
4. git commit
5. git push

没有完成这 5 步，不能算“好了”。

### 2.3 pm-coder 不得闷头全包
`pm-coder` 的默认动作应该是：
1. 定义小目标
2. 指派给合适 agent
3. 收结果
4. 决定下一小步

只有在任务极小、且明显不值得分发时，才允许自己直接动手。

### 2.4 避免反复 restart 同一个 agent
如果需要补充要求，优先：
- 看当前结果
- 基于结果下一个小步

不要通过反复 restart 把 agent 拉回“重新读上下文”的起点。

## 3. 标准执行链路

### 模式 A：正常多 agent 协作
1. `pm-coder` 定义一个小步目标
2. `pm-coder` 拉起 `code-helper` 落代码
3. 如涉及明显 UI 变化，再拉 `ui-designer` 做检查
4. 拉 `qa-tester` 跑 build / 回归
5. `pm-coder` 汇总结果
6. 完成 commit / push
7. `main` 对外汇报

### 模式 B：主控接管
如果发现以下任一情况，`main` 可直接接管当前轮：
- agent 被反复 restart，持续不出结果
- 多 agent 没有真正分发，只停留在口头协作
- 已经改了代码，但迟迟不 build / 不提交
- 任务规模很小，继续调度成本高于直接收尾

接管后仍要遵守：
- 小步目标
- build
- audit
- commit
- push

## 4. 实时汇报规则

每个小步至少汇报 3 次：
1. **开始时**：这一步做什么，谁来做
2. **中间态**：代码是否已改、build 是否开始
3. **结束时**：是否通过、commit 是什么、是否已 push

禁止只在最后一句说“做完了”。

## 5. 当前适用于 Recall 的推荐拓扑

- `pm-coder` → `code-helper`
- `pm-coder` → `ui-designer`
- `pm-coder` → `qa-tester`
- `main` 监控全部状态

当前不建议一上来开放所有 agent 全互调，避免任务流失控。

## 6. 当前问题复盘（2026-03-21）

本次流程卡住的直接原因：
- `pm-coder` 实际仍在单兵写代码
- 多次 steer 触发 restart，导致它反复回到“重读上下文 / 重写 ReviewPanel”
- 已写文件，但没有稳定推进到 build / audit / commit / push

后续必须按本文件流程执行，避免再次出现“代码改了但流程没收口”的假完成状态。
