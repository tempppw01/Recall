# Recall 项目整改清单（重构起步版）

> 目标：在不打断当前可用性的前提下，逐步完成工程化重构。
> 当前基线版本：`0.1`

## P0（立即处理，影响稳定性/可用性）

1. **统一版本基线**
   - 统一 `package.json`、`package-lock.json`、应用内展示版本、README 版本为 `0.1`。
   - 后续采用单一来源（建议以 `package.json` 为准，构建时注入前端）。

2. **修复数据库配置不一致（Prisma vs Docker）**
   - 当前 `prisma/schema.prisma` 使用 `postgresql`，但 Docker 默认 `DATABASE_URL=file:...`（SQLite 风格）。
   - 明确一条默认链路（建议默认 PostgreSQL，或改为真正 SQLite 二选一），避免“能构建不能运行”。

3. **梳理鉴权与动态 PG 连接边界**
   - 现状中通过 `x-pg-*` 请求头可进入动态连接分支，需确认是否可绕过标准登录流程。
   - 给出明确模式：
     - A) 单机本地模式（仅本地存储，不接收远程 DB 凭据）
     - B) 受控服务端模式（由服务端配置 DB，不信任客户端头）

## P1（近期处理，提升可维护性）

4. **拆分超大页面 `src/app/page.tsx`**
   - 按功能拆分：任务、日历、习惯、倒数日、AI、同步、设置。
   - 抽离 hooks：状态管理、筛选、日期处理、同步调度。
   - 目标：单文件控制在可维护范围（建议 < 800 行）。

5. **统一数据层边界（store 与 API 语义一致）**
   - 规范 `Task/Habit/Countdown` 的字段默认值与可选字段。
   - 明确本地写入、远端同步、冲突处理（LWW 或版本号）的规则。

6. **文档收敛**
   - 将无关或临时文档迁移到 `docs/archive/` 或删除。
   - `docs/ARCHITECTURE.md` 增补实际运行路径（本地-only / PG / Redis / WebDAV）。

## P2（中期处理，工程质量提升）

7. **增加基础测试**
   - 单测：日期/重复规则/提醒时间计算。
   - API 测试：tasks/habits/countdowns 的 CRUD 与异常分支。

8. **错误与可观测性**
   - 统一 API 错误结构（code/message/details）。
   - 增加关键路径日志与最小监控指标。

9. **发布流程标准化**
   - 明确镜像构建参数与运行时环境变量。
   - 增加 release checklist（版本、迁移、回滚点、变更摘要）。

---

## 建议的重构节奏（你可以按这个推进）

- **第 1 周**：先做 P0（版本统一 + DB 方案定稿 + 鉴权边界）
- **第 2 周**：拆 `page.tsx`（先拆状态与 hooks，不改 UI）
- **第 3 周**：补测试与错误体系
- **第 4 周**：收敛文档与发布流程

## 完成定义（DoD）

- 本地开发、Docker、基础 API 三条链路都能稳定跑通。
- 同步与鉴权行为可描述、可复现、可测试。
- 新人只看 README + docs 即可在 15 分钟内启动项目。
