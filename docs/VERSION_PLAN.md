# Recall 版本计划（基于当前代码状态）

> 当前版本：`0.1.1`
> 制定原则：先修可见问题，再收稳定性，再做结构重构，最后补工程质量。

---

## 总体策略

当前项目已经具备可运行雏形，但还处于 **“功能可用、结构未稳、工程质量待补”** 的阶段。

因此版本规划不建议一开始就冲 `1.0.0`，而应分成四段：

1. **0.1.x：修补可见问题**
2. **0.2.x：收敛运行链路与配置边界**
3. **0.3.x：做结构性重构**
4. **0.4.x：补测试、发布、可维护性**
5. **1.0.0：进入稳定发布阶段**

---

## v0.1.x —— UI / 发布 / 直接暴露问题修复期

### v0.1.1（当前）
目标：先把用户一眼能看到的问题修掉。

已包含 / 正在处理：
- 日历视图城市搜索结果重复
- 选择城市后再次触发搜索导致候选重复弹出
- Docker Hub 自动发布工作流
- Docker 多架构镜像发布（amd64 / arm64）

### v0.1.2
目标：再做一轮 UI 清理。

建议内容：
- 修复日历视图剩余布局问题（输入框、候选框、天气卡片状态一致性）
- 检查月视图 / 周视图 / 日视图切换时的状态残留
- 修复空态、按钮 hover、文本溢出、边距错位
- 检查移动端基础显示问题

### v0.1.3
目标：收拾开发脚本和基础发布体验。

建议内容：
- 修复 `npm run lint` / `npm run build` / `npm run start` 脚本兼容性
- 明确 Next.js 当前版本下的正确 CLI 用法
- 补 README 中的开发 / 构建 / Docker 发布说明
- 增加最小 release checklist

---

## v0.2.x —— 稳定性与边界收敛期

### v0.2.0
目标：把“能跑”变成“跑得明白”。

重点：
- 统一 Prisma / Docker / 环境变量的数据库方案
- 明确默认运行模式（本地模式 / 服务端模式）
- 收敛 `DATABASE_URL` 逻辑，避免 PostgreSQL / SQLite 混乱
- 梳理动态 PG 连接是否允许保留

### v0.2.1
目标：收鉴权边界。

重点：
- 检查 NextAuth 流程是否稳定
- 确认 `x-pg-*` 分支不会绕过正常用户边界
- 统一 API 未登录 / 异常返回格式

### v0.2.2
目标：把同步逻辑先规范起来。

重点：
- 明确 LocalStorage、PG、Redis、WebDAV 的职责边界
- 约定同步冲突策略（例如 LWW）
- 明确哪些字段是客户端真源，哪些是服务端真源

---

## v0.3.x —— 结构重构期

### v0.3.0
目标：拆掉当前最大的维护风险点。

重点：
- 拆分 `src/app/page.tsx`
- 分模块重组：
  - calendar
  - tasks
  - habits
  - countdowns
  - settings
  - ai
- 抽离 hooks：
  - useCalendar
  - useTasks
  - useWeather
  - useSync

### v0.3.1
目标：统一类型与数据边界。

重点：
- 收敛 Task / Habit / Countdown 类型定义
- 收敛 store 层与 API 层字段语义
- 减少页面直接操作复杂数据结构的情况

### v0.3.2
目标：把“UI 可跑”变成“模块可维护”。

重点：
- 增加 feature 级目录结构
- 降低页面耦合
- 减少巨型组件 props 传递

---

## v0.4.x —— 工程质量补齐期

### v0.4.0
目标：增加最基本的测试保障。

重点：
- 日期处理测试
- 重复规则测试
- 提醒时间计算测试
- tasks / habits / countdowns API 基础测试

### v0.4.1
目标：补错误处理与可观测性。

重点：
- 统一 API 错误结构
- 增加关键日志
- 增加同步失败、天气失败、AI 失败的可见提示

### v0.4.2
目标：完善发布质量。

重点：
- release checklist
- Docker 发布说明
- 环境变量说明补全
- 如果需要，再补 GitHub Release 自动化

---

## v1.0.0 —— 稳定版条件

只有在以下条件大体满足时，再进入 `1.0.0`：

- UI 层没有明显阻塞性问题
- 本地开发 / Docker / 基础部署链路稳定
- 数据存储方案明确且一致
- 鉴权边界清楚
- 关键模块已拆分，不再依赖超大单文件
- 至少具备基础测试与可回归能力
- 发布流程可重复、可追踪

换句话说：
**1.0.0 不是“功能很多”，而是“核心链路稳定”。**

---

## 建议的推进顺序

### 第一阶段（现在）
- 发 `0.1.1`
- 再补 `0.1.2`
- 再收 `0.1.3`

### 第二阶段
- 上 `0.2.0`：统一 DB 链路
- 再做 `0.2.1`：鉴权边界
- 再做 `0.2.2`：同步边界

### 第三阶段
- 上 `0.3.0`：拆 `page.tsx`

### 第四阶段
- 上 `0.4.x`：测试 + 质量 + 发布

---

## 一句话总结

如果按现在的实际情况来走，我建议：

- **短期别急着冲 1.0**
- **先把 0.1.x 做扎实**
- **0.2.x 解决“能不能稳定跑”**
- **0.3.x 解决“后面还能不能继续维护”**
- **0.4.x 再解决“敢不敢稳定发版”**

这条路线比较稳，也比较符合你现在“先一点点重构”的节奏。
